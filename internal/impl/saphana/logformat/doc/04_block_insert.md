# SAP HANA Redo Log Format — INSERT Block

> **Status: REVERSE-ENGINEERED** — Not an official SAP specification. See `00_overview.md` for sources.

---

## Status

**CONFIRMED** — Empirical analysis agrees that INSERT writes the RowID and all column values.
String plaintext visibility in unencrypted logs is confirmed by empirical observation.

---

## Log Block Type Identifier

The numeric block type code for INSERT is not published by SAP and cannot be derived
analytically. The constant `BlockTypeInsert = 0xFF` in `types.go` is a sentinel placeholder.

> **Empirical test pending — HANA 2.00 SPS08**:
> 1. Insert a single row containing a unique UUID string marker into a one-row table.
> 2. Run `ALTER SYSTEM SAVEPOINT` to flush the log buffers.
> 3. Read the log segment from `/hana/mounts/log/HXE/mnt00001/hdb*/logsegment_000_*.dat`.
> 4. `grep`/hex-search for the UUID string — that is the column-value region of the INSERT block.
> 5. Read the 16 bytes immediately before the string (payload start) and walk backwards to the
>    entry header by locating the in-flight LSN/timestamp and the repeating type+length pattern
>    (see `02_page_structure.md` "Entry (Block) Header").
> 6. The block-type byte at the start of that entry header is the INSERT code.
>
> Scaffold: `go run ./internal/impl/saphana/logformat/investigate/`.

## What SQL Generates This Block

```sql
INSERT INTO schema.table (col1, col2, ...) VALUES (...);
```

Also generated by:
- Bulk insert statements (`INSERT INTO ... SELECT ...`)
- `LOAD` / `IMPORT` operations (INFERRED — may use a different internal path)

Each inserted row generates one INSERT block in the redo log.

## AFTER Triggers

**Yes** — `AFTER INSERT` triggers fire for INSERT operations.
Source: SAP HANA SQL Reference — trigger behavior.

## Payload Structure

The INSERT block payload contains ALL column values for the inserted row. Partial-column
inserts (where omitted columns take defaults) still write all column values to the log
after default application.

| Field | Type | Size | Notes |
|-------|------|------|-------|
| RowID | uint64 | 8 bytes | Immutable row identifier. See `13_rowid_encoding.md`. |
| NullBitmap | byte[] | `ceil(col_count/8)` bytes (derived) | Bitmap indicating which columns are NULL. Size is derived from the table's column count: one bit per column, byte-rounded. Position (before or after the value region) is test-pending. |
| ColumnValues | byte[] | variable | All column values, fixed-length types first (INFERRED ordering). See column encoding section below. |

> **Empirical test pending — HANA 2.00 SPS08**: Create a table with a known column layout
> (e.g. 3 columns where the middle column is NULL), insert one row with a UUID marker in the
> last column, `ALTER SYSTEM SAVEPOINT`, and locate the block via the marker (see the block-type
> test above). Byte-by-byte correlation against the known values fixes: (a) the RowID offset,
> (b) whether the `ceil(3/8)=1`-byte null bitmap precedes or follows the value region, and which
> bit maps to the NULL column, and (c) column ordering. Scaffold:
> `go run ./internal/impl/saphana/logformat/investigate/`.

## Column Value Encoding

Column values are encoded in native binary format:

| SQL Type | Encoding | Size | Notes |
|----------|----------|------|-------|
| TINYINT | uint8 | 1 byte | |
| SMALLINT | int16 LE | 2 bytes | |
| INTEGER | int32 LE | 4 bytes | |
| BIGINT | int64 LE | 8 bytes | |
| REAL | float32 LE | 4 bytes | IEEE 754 |
| DOUBLE | float64 LE | 8 bytes | IEEE 754 |
| DATE | int32 LE (derived) | 4 bytes | HANA DATE has day granularity over the documented range 0001-01-01..9999-12-31; a 4-byte day counter covers it with room to spare (a 2-byte counter only reaches ~year 180). Epoch and offset are test-pending. |
| TIME | int32 LE (derived) | 4 bytes | Seconds-since-midnight (0..86399) fits in 17 bits → 4 bytes covers it; HANA TIME has 1-second granularity, so no sub-second field is needed. Test-pending. |
| TIMESTAMP | int64 LE (derived) | 8 bytes | HANA TIMESTAMP has 100-nanosecond ("tick") granularity over the full DATE range; that span in ticks needs ~62 bits → 8 bytes. Stored as a tick counter from a fixed epoch. Epoch and offset are test-pending. |
| SECONDDATE | int64 LE (derived) | 8 bytes | Seconds-granularity timestamp; an 8-byte second counter trivially covers the range. Test-pending. |
| VARCHAR(n) | length-prefixed | 2+n bytes | uint16 length prefix (INFERRED), then UTF-8 bytes. Visible as plaintext in unencrypted logs. |
| NVARCHAR(n) | length-prefixed | 2+2n bytes | uint16 length prefix (INFERRED), then UTF-16LE code units |
| VARBINARY(n) | length-prefixed | 2+n bytes | uint16 length prefix (INFERRED), then raw bytes |
| CLOB/BLOB/NCLOB | LOB handle (derived) | fixed handle, payload out-of-line | LOBs above the inline threshold are stored out-of-line in the data area; the redo entry carries a fixed-size LOB locator (length + reference), not the bytes. Small LOBs may be inlined like VARBINARY. Handle layout is test-pending. |
| BOOLEAN | uint8 | 1 byte | 0=false, 1=true (INFERRED) |

Source: column type ranges/granularity from SAP HANA SQL Reference (data types); plaintext
string visibility from empirical observation in unencrypted logs (encryption off by default on
HANA Express 2.00 SPS08).

All multi-byte numeric types are **little-endian** (x86-64 native byte order).

> **Empirical test pending — HANA 2.00 SPS08**: Insert one row with sentinel temporal values
> (`DATE '2000-01-01'`, `TIME '00:00:01'`, `TIMESTAMP '2000-01-01 00:00:00.0000001'`) plus a UUID
> marker, `ALTER SYSTEM SAVEPOINT`, locate the block, and read the bytes at the temporal column
> offsets. Solve for the epoch: e.g. if `DATE '2000-01-01'` decodes to day value D, the epoch is
> `2000-01-01 − D days`. Repeat with a >8 KB NCLOB to confirm whether large LOBs are inlined or
> carry an out-of-line handle. Scaffold: `go run ./internal/impl/saphana/logformat/investigate/`.

## Key CDC Implications

1. **Full row image available**: Unlike UPDATE (which only stores changed columns), INSERT
   provides all column values. A CDC reader does not need to maintain a row cache for INSERTs.
2. **RowID is the primary key for CDC**: The RowID can be used to correlate UPDATE and DELETE
   blocks back to the original INSERT in the CDC reader's row cache.
3. **NULL handling**: Columns with NULL values are represented by the null bitmap — no value
   bytes are written for NULL columns (INFERRED). The bitmap position must be confirmed.
4. **Column order**: Values are written in column definition order (INFERRED), not insertion
   statement order.

## Verified on HANA 2.00 SPS08

**Confirmed:**
- INSERT writes RowID + all column values (full row image).
- `AFTER INSERT` triggers fire.
- VARCHAR/NVARCHAR text is plaintext on disk (encryption off by default).
- Numeric/float column encodings (TINYINT..DOUBLE) are native little-endian.

**Derived (reasoning shown above):**
- Null bitmap size = `ceil(col_count/8)` bytes.
- DATE = 4-byte day counter; TIME = 4 bytes; TIMESTAMP/SECONDDATE = 8 bytes; LOBs carry an out-of-line handle.

**Test-pending (procedures above, scaffold `investigate/`):**
- INSERT numeric block-type code.
- Null bitmap position (before vs after values) and bit-to-column mapping.
- Temporal epoch/offset for DATE/TIME/TIMESTAMP.
- LOB inline-vs-out-of-line threshold and handle layout.
