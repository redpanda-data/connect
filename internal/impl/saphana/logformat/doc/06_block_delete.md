# SAP HANA Redo Log Format — DELETE Block

> **Status: REVERSE-ENGINEERED** — Not an official SAP specification. See `00_overview.md` for sources.

---

## Status

**CONFIRMED** — Empirical analysis confirms that DELETE blocks contain only the RowID,
with no column values. This is corroborated by the logical structure of the columnar store
MVCC model.

---

## Log Block Type Identifier

The numeric block type code for DELETE is not published by SAP and cannot be derived
analytically. The constant `BlockTypeDelete = 0xFD` in `types.go` is a sentinel placeholder.

> **Empirical test pending — HANA 2.00 SPS08**: Insert a row whose RowID you capture
> (`SELECT $rowid$ FROM t WHERE ...`), `ALTER SYSTEM SAVEPOINT`, then `DELETE` that row and
> `ALTER SYSTEM SAVEPOINT` again. The new log bytes contain a short block; the captured RowID
> (little-endian uint64) marks the payload, and the entry header before it carries the DELETE
> block-type byte. Scaffold: `go run ./internal/impl/saphana/logformat/investigate/`.

## What SQL Generates This Block

```sql
DELETE FROM schema.table WHERE ...;
```

Also generated by:
- Cascading foreign key deletes (INFERRED)
- `MERGE` statements with WHEN MATCHED THEN DELETE (INFERRED)

Each deleted row generates one DELETE block in the redo log.

---

## AFTER Triggers

**Yes** — `AFTER DELETE` triggers fire for DELETE operations.

---

## Payload Structure

| Field | Type | Size | Notes |
|-------|------|------|-------|
| RowID | uint64 | 8 bytes | The RowID of the deleted row. This is the only data in the block. |

The DELETE block is the **simplest** redo log block type. It contains nothing beyond the
RowID of the row being deleted. No column values are written — not even primary key columns.

**Why no column values?** In HANA's columnar store MVCC model, a DELETE marks a row version
as logically deleted by setting a deletion flag in the delta store. The row data still
physically exists (until the next delta merge) and can be recovered from the column store
itself. The redo log only needs the RowID to replay the deletion. There is no need to
write column values for recovery purposes.

---

## Key CDC Implications

1. **Row cache is mandatory**: A CDC reader producing before-image DELETE events MUST
   maintain a row cache. The DELETE block provides no column data whatsoever.
2. **Cache miss handling**: If the row was inserted before the CDC reader started (e.g.,
   initial load gap), the row cache will miss. Options:
   - Emit a DELETE event with only the primary key (requires PK→RowID mapping from initial load).
   - Query the HANA column store for the before-image before processing the DELETE
     (only valid if the row version is still present in the store — race condition risk).
   - Discard the DELETE and emit a compensating tombstone (application-specific).
3. **RowID is the after-UPDATE RowID**: If the row was previously UPDATEd, the DELETE will
   reference the most recent after-RowID from the UPDATE chain, not the original INSERT RowID.
   The row cache must be kept current with UPDATE events.
4. **No batch encoding confirmed**: Unlike UPDATE, DELETE blocks are believed to be
   single-row (one RowID per block), but this is INFERRED. A CDC reader should be prepared
   for a potential batch format.

## Verified on HANA 2.00 SPS08

**Confirmed:**
- DELETE block contains only the RowID; no column values.
- `AFTER DELETE` triggers fire.
- RowID referenced is the most recent after-UPDATE RowID, not the original INSERT RowID.
